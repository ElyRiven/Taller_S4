| [US-001](https://github.com/ElyRiven/Taller_S4/blob/main/US-001_ORIGINAL.md) | [US-001](https://github.com/ElyRiven/Taller_S4/blob/main/US-001_REFINED.md) | 1. El original no especifica qué servicios ni qué técnicas usar. El refinado nombra los 3 componentes (Frontend, Producer API, Consumer Worker), las técnicas concretas (`multi-stage`, usuario no-root) y la herramienta de escaneo (Trivy). 3. INVEST `Negotiable`: el original dice "el enfoque puede adaptarse". El refinado cierra la negociación documentando las herramientas acordadas (Trivy, GitHub Actions, `appuser`). 4. INVEST `Small`: el original penaliza con justificación vaga ("puede requerir dividirse"). El refinado sube a 3 porque el alcance quedó acotado a 3 componentes del MVP. 5. INVEST `Testable`: el original valida con "linters y pruebas de build" (genérico). El refinado especifica que QA puede forzar imágenes vulnerables para validar el bloqueo del pipeline. 6. El original tiene 1 escenario Gherkin que mezcla seguridad y eficiencia. El refinado descompone en 3 escenarios independientes: Multi-stage, No-Root y Bloqueo CI/CD, cada uno con criterio de fallo claro. 7. El DoD original valida con "linters y escáner de vulnerabilidades" sin especificar herramientas. El refinado agrega un ítem explícito de QA: workflow de GitHub Actions con Trivy y reglas de bloqueo diferenciadas por severidad (CRÍTICA/ALTA vs MEDIA/BAJA). |
| [US-002](https://github.com/ElyRiven/Taller_S4/blob/main/US-002_ORIGINAL.md) | [US-002](https://github.com/ElyRiven/Taller_S4/blob/main/US-002_REFINED.md) | 1. El original es vago ("despliegues consistentes y reproducibles"). El refinado agrega el rol DevOps, incorpora "orden de encendido confiable" como beneficio concreto. 3. INVEST `Negotiable`: el original dice "se puede ajustar la estructura". El refinado documenta la decisión tomada: se excluyeron límites de CPU/RAM por no ser producción. 4. INVEST `Estimable`: el original es genérico ("alcance claro"). El refinado nombra los dos entregables concretos (healthchecks + `.env`), lo que hace la estimación verificable. 5. INVEST `Testable`: el original dice "se valida con despliegues en cada ambiente". El refinado acota: se automatiza el levante y se verifica el orden de los healthchecks. 6. El original tiene 1 escenario Gherkin genérico sobre despliegue reproducible. El refinado tiene 3 escenarios: gestión de credenciales via `.env`, orden de encendido con healthchecks, y paridad entre ambientes `dev` y `test`. 7. El DoD original no menciona seguridad de credenciales ni healthchecks. El refinado agrega: verificación de comunicación entre contenedores en Smoke Tests, prueba de caída/recuperación en regresión, y un ítem de QA con `.env.example` versionado como plantilla para el equipo. |
| [US-003](https://github.com/ElyRiven/Taller_S4/blob/main/US-003_ORIGINAL.md) | [US-003](https://github.com/ElyRiven/Taller_S4/blob/main/US-003_REFINED.md) | 1. La refinada agrega una sección introductoria que no existe en el original, explicando explícitamente por qué la historia entra como WIP (limitaciones de tiempo) y cómo se gestionará de forma incremental. 2. La refinada agrega la subsección §3.1 con el contrato explícito de API (estructura JSON de `PagedKudoResponse`). El original solo menciona la eliminación de `Page` sin definir qué la reemplaza. 3. El original en INVEST `Valuable` incluye "escalado en lectura independiente **del escalado de escritura**". La refinada recorta esa frase. El resto de justificaciones INVEST son idénticas. 4. El original tiene 1 escenario Gherkin que solo cubre el flujo Query (GET). La refinada tiene 6 ACs que cubren: Query happy path, Command happy path, Query sad path, Command sad path (DLQ), Resiliencia ante caída de BD, y Testing de integración. 5. El original ignora completamente el flujo Command (RabbitMQ). La refinada lo cubre con AC 2 (happy path) y AC 4 (sad path / DLQ). 6. El original no menciona qué ocurre si la BD cae. La refinada agrega AC 5: la aplicación no debe fallar y los mensajes deben mantenerse en cola hasta que el servicio levante. 7. El original no define cómo validar el refactor en CI. La refinada agrega AC 6 con criterios concretos: Testcontainers, `RabbitTemplate`, consulta a `KudoRepository` y verificación de enrutamiento a DLQ. 8. El original usa `Dado/Cuando/Entonces` en español. La refinada usa `Given/When/Then/And` en inglés. 9. El resultado esperado del escenario original dice "listado idéntico al provisto por el anterior monolito" (ambiguo). La refinada lo reemplaza por "JSON definido en el contrato `PagedKudoResponse`" (verificable). 10. El original tiene 4 ítems de DoD genéricos con bullets. La refinada tiene 9 ítems con checkboxes (`[ ]`), agregando: aislamiento del DTO, Contract Test, validación del contrato §3.1, configuración de DLQ, tolerancia a caída de BD y cobertura de happy/sad path del consumer. |
