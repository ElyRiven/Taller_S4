Historia de Usuario 001
----

| ID Caso IA | Caso de Prueba generado por la Gema                            | Ajuste realizado por el probador                                                                                                       | ¿Por qué se ajustó?                                                                                       |
| ---------- | -------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------- |
| TC_001     | Validación de Arquitectura Multi-stage en Componentes de Kudos | Se añadió validación de que la imagen final no contenga JDK, Node ni herramientas de build y se definió un criterio objetivo de tamaño | La IA asumió que múltiples `FROM` garantizan multi-stage correcto, lo cual puede generar falsos positivos |
| TC_002     | Ejecución de Contenedores con Usuario No-Root (appuser)        | Se agregó validación de continuidad del servicio tras el bloqueo de permisos y se explicitó que el contenedor debe estar en ejecución  | La IA no consideró continuidad operativa ni dejó claras las precondiciones técnicas                       |
| TC_003     | Bloqueo de PR por Vulnerabilidades Críticas en Pipeline CI/CD  | Se separó el comportamiento esperado para vulnerabilidades CRITICAL y HIGH y se definió una dependencia vulnerable controlada          | La IA mezcló severidades sin diferenciar impacto real en el pipeline                                      |
| TC_004     | Continuidad de Pipeline con Vulnerabilidades de Bajo Impacto   | Se añadió validación explícita de logging y trazabilidad de vulnerabilidades LOW/MEDIUM                                                | La IA no garantizaba visibilidad del riesgo, solo la continuidad del pipeline                             |

Historia de Usuario 002
----

| ID Caso IA | Caso de Prueba generado por la Gema           | Ajuste realizado por el probador                                                                                                           | ¿Por qué se ajustó?                                                                                                     |
| ---------- | --------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------ | ----------------------------------------------------------------------------------------------------------------------- |
| TC_005     | Inyección Dinámica de Secretos (.env)         | Se agregó la validación de que las variables resueltas no queden expuestas en logs del arranque ni en el output de `docker-compose config` | La Gema validó solo la ausencia de texto plano en YAML, pero no consideró el riesgo de exposición indirecta de secretos |
| TC_006     | Control de Secuencia de Inicio (Healthchecks) | Se añadió la verificación de reintentos controlados y backoff en caso de que la dependencia tarde más de lo esperado en estar healthy      | La Gema asumió disponibilidad lineal de dependencias y no cubrió escenarios de latencia o arranque lento                |
| TC_007     | Paridad Funcional entre Ambientes Dev y Test  | Se incorporó la validación de variables de entorno y flags de configuración además de versiones de imágenes                                | La Gema limitó la paridad a versiones de software, omitiendo diferencias frecuentes de configuración entre ambientes    |
| TC_008     | Disponibilidad de Plantilla .env.example      | Se añadió la validación de que el archivo no sea referenciado por error en ningún `docker-compose` ni pipeline CI                          | La Gema no contempló el riesgo de uso accidental de `.env.example` en ejecución real                                    |

Historia de Usuario 003

| TC_009     | Validación de Desacoplamiento de Paginación (Capa Application/Domain) | Se añadió la verificación de que no existan dependencias indirectas (transitivas) a Spring Data mediante análisis de dependencias del build | La Gema solo validó imports directos, pero no contempló dependencias transitivas que rompen Clean Architecture |

| TC_012     | Resiliencia ante Indisponibilidad de Base de Datos                    | Se añadió la validación de política de reintentos y backoff para confirmar que no exista consumo agresivo de recursos                       | La Gema validó la no pérdida de mensajes, pero no evaluó impacto en estabilidad del servicio                   |
| TC_013     | Validación de Contrato de API (Contract Testing)                      | Se agregó la validación de versionado del contrato y compatibilidad hacia atrás (backward compatibility)                                    | La Gema verificó solo el contrato actual, sin considerar evolución del API y ruptura de consumidores           |
